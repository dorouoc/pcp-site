1. Project Overview

PUMA (PUMA Understanding and Management with Agents) is a research project investigating whether autonomous Large Language Model agents can perform practical software project management tasks — specifically issue triage and effort estimation — with accuracy and reproducibility comparable to human experts.

The project is conducted as a Bachelor's / Master's thesis in ICT Engineering and uses a multi-stage empirical pipeline: systematic literature review → artifact design → LLM-based agent construction → experiment on real-world datasets (TAWOS, Jira SR) → statistical validation and replication package publication.

Core Mission

Design, implement, and empirically evaluate a multi-agent LLM system — the PUMA platform — that autonomously triages software issues (classifying type, priority, and component) and estimates story-point effort from issue text. The platform is benchmarked against human-annotated ground truth using standard SE metrics (accuracy, F1, MAE, MRE) and statistical tests (Wilcoxon, ANOVA).

Research Hypotheses

  • H1 (Triage): An LLM agent using few-shot prompting achieves >75% F1 on issue type and priority classification on the TAWOS dataset.
  • H2 (Estimation): An LLM agent using chain-of-thought reasoning achieves a Mean Relative Error ≤35% on story-point estimation on the Jira SR dataset.

Datasets

TAWOS

The Agile Work Items and Operations Study. A curated dataset of Jira issues from multiple open-source projects, providing ground-truth labels for type, priority, component, and resolution. Hosted on GitHub (SOLAR-group/TAWOS).

Jira SR (Story Points)

Zenodo dataset (DOI: 10.5281/zenodo.5901893) containing story point estimates and issue descriptions from multiple industrial and open-source Jira projects, enabling regression and classification experiments on effort estimation.

Technology Stack

Claude (Anthropic) GPT-4o (OpenAI) Llama 3 (local) Ollama LangGraph AutoGen CrewAI Python Zotero Obsidian GitHub Actions Prometheus / Grafana Semantic Scholar API Promptfoo PydanticAI Supabase

2. Knowledge Management Methodologies

The vault fuses six complementary knowledge management systems into a single coherent workspace. Each system is not used in isolation — they reinforce each other, each covering a different dimension of the research lifecycle.

MOC — Maps of Content

MOC notes act as index pages that gather links to related notes by topic, forming a navigable graph of knowledge rather than a rigid hierarchy. Each major domain (agents, benchmarks, methods, tools, videos) has a dedicated MOC in 80 - MOC/81 Topic-Maps/. The MOC layer is the primary navigation surface for the vault; it replaces folders as the main organizational paradigm at the semantic level.

GTD — Getting Things Done

David Allen's GTD system governs all task and commitment management. The vault's 90 - GTD/ area houses the active task list, master projects list, and someday/maybe queue. Daily and weekly reviews (templated) ensure that captured tasks are processed, clarified, and either acted upon, deferred, or discarded. GTD prevents the research inbox from becoming a graveyard of unprocessed ideas.

PARA — Projects · Areas · Resources · Archive

Tiago Forte's PARA framework provides the macro-level organizational skeleton. 40 - Projects holds active, deadline-bound research work. 50 - Areas contains ongoing responsibilities with no end date (research practice, writing, ethics). 60 - Resources stores reusable assets (prompts, workflows, glossary). 70 - Archive holds completed and deprecated material. PARA ensures every note has a home and that homes decay gracefully as work evolves.

Zettelkasten

Niklas Luhmann's slip-box methodology generates atomic, permanent, densely-linked knowledge notes in 30 - Permanent/. Each permanent note encodes a single idea, carries a unique identifier, and links to related notes by concept rather than by topic. The Zettelkasten is the intellectual core of the vault — it is where raw literature notes are distilled into original insights, conceptual clusters emerge, and theoretical frameworks crystallize. Literature notes (LN-) feed the Zettelkasten; permanent notes (PN-) are its output.

Johnny Decimal

A numeric address system that assigns a unique two-digit code to every area (00–90) and a decimal sub-code to every sub-area (20.1, 20.2, …). This prevents folder sprawl, enables precise cross-referencing (e.g., see 60.64 Glossary), and makes every file findable by its address alone. The entire vault is structured under Johnny Decimal, from root folders to sub-sub-directories.

SDD / BDD / BMAD

Spec-Driven Development (SDD) treats formal specifications as the source of truth from which code and experiments are derived. Behaviour-Driven Development (BDD) expresses requirements as human-readable scenarios. BMAD (Build, Manage, Adapt, Deploy) is an AI-native project management methodology that orchestrates specialized AI agents through clearly defined roles (Analyst, Architect, QA, etc.) using structured PRDs and sprint boards. The PUMA platform's own development follows SDD+BMAD, treating the AI agent system as a product developed by AI agents.

3. Research Frameworks

PUMA integrates a layered stack of research methodologies — from the macro study design down to the micro level of individual prompting strategies. Together they ensure rigor, reproducibility, and systematic knowledge generation.

Primary Research Methods

EBSE + SLR / PRISMA

Evidence-Based Software Engineering (EBSE) is the epistemological foundation: research claims must be grounded in empirical evidence from controlled experiments or systematic reviews. The Systematic Literature Review (SLR) protocol — reported using the PRISMA 2020 checklist — maps the state of the art on LLM agents in project management, defining inclusion/exclusion criteria, search strings, screening stages, and a PRISMA flow diagram.

DSR — Design Science Research

Hevner et al.'s DSR framework treats the PUMA agent platform as an artifact to be designed, built, and evaluated against utility criteria. The DSR cycle (problem identification → design → artifact construction → evaluation → communication) maps directly to the thesis chapters. DSR legitimizes the construction of a novel AI system as a valid research contribution, not merely an engineering exercise.

Grounded Theory

Used for qualitative analysis of emergent patterns — particularly for analysing agent reasoning traces, failure modes, and unexpected behaviours during experiments. Constant comparison, open coding, and memo-writing allow conceptual categories to emerge from the data rather than being imposed a priori. Especially relevant for Stage 2 (estimation) where LLM chain-of-thought outputs require qualitative interpretation.

MIT Student Method

A structured research workflow from the MIT AI Lab (Working Paper WP316, Patrick Winston). Emphasises vision → steps → news as the three essential contributions of any research work. Applied in PUMA to structure each experimental stage: articulate the novel contribution, identify the incremental steps, and frame results as publishable findings. Literature notes follow a three-pass reading protocol (Keshav 2007) within this framework.

Prompting & Agent Engineering Frameworks

FrameworkCategoryDescription
RCOIFPromptingRole · Context · Objective · Instructions · Format. A structured prompting template that eliminates ambiguity by forcing explicit specification of each element before querying an LLM. Used as the default prompt format across the vault.
EGIPromptingExplicit Goal-setting and Iterative refinement. A dialogue-based prompting loop where each agent turn either confirms goal alignment or requests clarification before proceeding. Minimises hallucinated assumptions.
AMIPromptingAnchored Multi-step Inference. Grounds long reasoning chains in verified anchors (retrieved facts, defined terms) at each step to prevent reasoning drift in multi-hop questions.
DRCAPromptingDecompose · Reason · Critique · Answer. A four-phase prompt pattern that forces explicit task decomposition, step-by-step reasoning, self-critique, and a final synthesized answer. Particularly effective for complex triage decisions.
IIPRPromptingIntent · Information · Process · Result. A structured template for research prompts ensuring the LLM understands the intent behind a query, the information context available, the required process, and the expected result format.
CoTPromptingChain-of-Thought prompting. Elicits step-by-step reasoning from LLMs by inserting the phrase "Let's think step by step" or by providing reasoning exemplars. Dramatically improves performance on multi-step tasks like story-point estimation.
Few-ShotPromptingProvides 3–8 labelled examples in the prompt context, enabling in-context learning without model fine-tuning. Used in Stage 1 (triage) to demonstrate the labelling schema to the LLM.
Zero-Shot CoTPromptingCombines zero-shot prompting with chain-of-thought elicitation. The model reasons without examples, guided only by the instruction to think step by step. Baseline for comparison against few-shot variants.
CDDDevelopmentContext-Driven Development. Treats the LLM context window as the primary design space. Every agent, tool, and memory component is designed with explicit consideration of context window limits, injection order, and compression strategies.
Agent OSArchitectureA framework for orchestrating multiple specialized LLM agents through a shared operating-system-like interface: task queue, memory bus, tool registry, and inter-agent messaging. PUMA implements a lightweight Agent OS with Claude Code as the orchestrator runtime.
Agent Prompt EngineeringArchitectureThe discipline of designing, testing, and iterating on system prompts for autonomous agents. Includes techniques such as role priming, skill injection, negative example provision, and exit-condition specification. The 60 - Resources/61 Prompts/ area is the prompt engineering library.

5. The .claude Folder & Skills

The .claude/ folder is the AI integration layer of the vault. It contains two sub-systems: the PUMA skill library (custom Claude AI skills written specifically for this research project) and the obsidian-skills plugin (a community marketplace of Obsidian-specific Claude skills). Together they turn Claude Code into a domain-aware research assistant that understands vault conventions, research methodologies, and Obsidian internals.

How Skills Work

Each skill is a Markdown file (named *.md or SKILL.md) containing a name, description, and a body that provides Claude with structured domain knowledge. When Claude operates within this vault, it automatically loads relevant skills based on the task at hand — a governance layer (puma-core) is always active; domain skills are injected as needed. Skills eliminate the need to repeat context in every prompt: instead of explaining "the vault uses Johnny Decimal and the note prefix is LN-", Claude already knows this from the skill system.
